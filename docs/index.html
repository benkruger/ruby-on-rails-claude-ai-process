GitHub Changelog

FLOW

An opinionated 8-phase development lifecycle for Claude Code — built for Ruby on Rails. Every feature follows the same phases in the same order. No shortcuts.

Step 1: Install in Claude Code

/plugin marketplace add benkruger/flow
/plugin install flow@flow-marketplace

Step 2: Start a feature in your Rails project

/flow:start user authentication

FLOW creates a git worktree, upgrades your gems, opens a GitHub PR, runs bin/ci to establish a green baseline, and lands you in Phase 2: Research.

The Problem

Claude Code is powerful, but undisciplined by default.

FLOW imposes structure. Not bureaucracy — discipline.

The Pipeline

Every feature goes through 8 phases. Each phase has mandatory entry guards, structured outputs written to a persistent state file, and back-navigation when something was missed. No phase can be skipped.

1
Start Worktree · bundle update · bin/ci baseline · PR created
2
Research Full class hierarchy · callbacks · test/support/ · risks documented sub-agent
3
Design 2–3 alternatives validated · user picks one · approved before any code sub-agent
4
Plan Task feasibility verified · TDD ordering · section-by-section approval sub-agent
5
Code Test-first per task · diff review · bin/ci gate · commit per task
6
Review Design alignment · research risk coverage · Rails anti-patterns sub-agent
7
Reflect Corrections become CLAUDE.md patterns · plugin gaps noted
8
Cleanup Worktree removed · state file deleted · feature done

Phases 2, 3, 4, and 6 use mandatory Explore-type sub-agents to read the codebase. The main conversation stays focused on decisions. Sub-agents handle all file reads and return structured findings — keeping the main context clean for decision-making throughout a long session.

Architecture

Sub-Agent Architecture

Research, Design, Plan, and Review phases launch Explore-type sub-agents. By the time Code begins, every callback has been found, every risk documented. Code trusts the state file — it doesn’t re-explore.

State File Persistence

Every feature has a .claude/flow-states/<branch>.json file tracking research findings, design decisions, plan tasks, captured notes, and per-phase timing. State survives session breaks and compaction.

Session Hook — Auto-Resume

Every session start — new terminal, /clear, /compact — triggers a hook that scans for in-progress features. Claude’s first action is /flow:resume. Close your laptop, open Claude Code the next morning, pick up exactly where you left off.

The Learning Pipeline

User corrects Claude → /flow:note captures it as a reusable pattern → Reflect synthesizes all notes → each approved pattern added to CLAUDE.md → every future feature benefits. The learnings compound.

Phase Back-Navigation

Not a linear checklist. If Design reveals a gap in Research, go back. If Code reveals a flaw in the Plan, go back. Prior findings are preserved and extended — never discarded. Later phases are invalidated and re-approved.

100% Coverage Enforced

Code cannot transition to Review until coverage/uncovered.txt is empty. bin/ci must be green before every commit and every phase transition. Technical debt is blocked at the source.

Instructions Are Advisory. Gates Aren’t.

Most agent workflows put enforcement in instructions: “always run bin/ci”, “never skip Research”. Instructions work until they don’t. FLOW’s phase enforcement is layered and deterministic. There is no instruction path from an incomplete phase to the next one running.

Inline Phase Guard

Every phase skill opens with a Python gate that reads the state file and exits immediately with BLOCKED if the previous phase isn’t complete. The skill doesn’t run — there’s nothing for Claude to interpret or override.

check-phase.py

A standalone verification script callable from anywhere in the workflow. One source of truth for phase state, shared by skills, hooks, and utility commands alike. No duplication, no drift.

SessionStart Hook

Fires on every session start — new terminal, /clear, /compact. Reads the state file and injects the current phase directly into Claude’s context. After a week away, Claude opens knowing exactly where it is and cannot proceed as if it doesn’t.

What It Enforces

  • Worktree isolation — main is never touched directly; multiple features run in parallel
  • Research before design — full class hierarchy read, callbacks found, risks documented first
  • Design alternatives required — 2–3 distinct approaches validated before user picks one
  • TDD always — test must fail before implementation is written; test must pass before commit
  • bin/ci gate — must be green before every commit and every phase transition
  • No disabling RuboCop — fix the code, not the cop; no # rubocop:disable comments
  • Commit discipline — imperative verb + tl;dr + per-file breakdown, every commit

Utility Commands

Available at any point in the workflow, regardless of phase.

CommandWhat it does
/flow:commitFull diff review, approved commit message, pull before push
/flow:statusCurrent phase, PR link, cumulative time per phase, next step
/flow:resumeRe-asks last transition question; rebuilds full context on new session
/flow:noteCaptures corrections to state file — auto-invoked when Claude is wrong
/flow:releaseBump version, tag, create GitHub Release (maintainer use)

Part of the Ecosystem

FLOW is part of a growing community of disciplined Claude Code plugins. Two projects worth knowing that inspired and motivated me:

metaswarm by Dave Sifry. A multi-agent orchestration framework with 18 specialized agents, cross-model adversarial review, and full pipeline orchestration from GitHub issue to merged PR. If FLOW is disciplined Rails development, metaswarm is disciplined development at scale.

Superpowers by Jesse Vincent. Foundational agentic skills for Claude Code including TDD, systematic debugging, and plan writing. Proved that disciplined agent workflows are not overhead — they’re what make autonomous development reliable.
