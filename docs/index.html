GitHub Changelog

FLOW

An opinionated 8-phase software development lifecycle for Claude Code. Every feature follows the same phases in the same order. No shortcuts. Supports Rails and Python.

Why

Claude Code is powerful, but undisciplined by default. Left to itself, it writes code, skips tests, misses edge cases, and moves on. FLOW imposes structure. Not bureaucracy — discipline.

FLOW is for the engineer who wants to stay in the driver’s seat: who decides what gets built, approves the design, reviews the plan, and merges the result. You control the autonomy — fully manual, fully autonomous, or anywhere in between, per skill. You remain the engineer. Claude does the work.

How

Every feature goes through 8 phases. Each phase must complete before the next one starts. If something was missed, go back.

1
Start Branch, PR, green CI baseline, dependencies upgraded
2
Plan Explore the codebase, design the approach, identify risks, write tasks
3
Code Test first, then implement, review every diff, commit per task
4
Simplify Refactor for clarity, auto-commit
5
Review Check plan alignment, verify risk coverage, catch bad patterns
6
Security Scan for vulnerabilities, auth gaps, data exposure, injection
7
Learning Capture what went wrong, route corrections to permanent homes
8
Cleanup Clean up, delete state, done

Each phase is a single command: /flow-start, /flow-plan, /flow-code, /flow-simplify, /flow-review, /flow-security, /flow-learning, /flow-cleanup.

Zero dependencies — pure Markdown skills with a thin Python dispatcher. Opus for planning, code, and security. Sonnet for review. Haiku for setup. Plan mode for codebase exploration. .flow-states/<branch>.json tracks everything — state survives session breaks and compaction.

The Learning System

Most AI coding tools forget everything when the session ends. FLOW captures corrections as they happen, rescues session memory before worktrees are destroyed, and routes each learning to the right permanent home — where it compounds across every future feature.

1
Capture User corrects Claude → /flow-note captures it automatically
2
Observe Claude writes patterns and observations to auto-memory during feature work
3
Synthesize Learning reads three sources: CLAUDE.md rules · conversation context · state/plan data
4
Route Each learning is routed to its correct permanent destination
5
Survive Worktree is destroyed — nothing is lost

5 Destinations

Global CLAUDE.md

Process rules that apply to all projects. “Always run bin/ci before committing.”

Project CLAUDE.md

Project-specific architecture. “Callbacks set fields from Current — don’t pass explicitly.”

Global Rules

Universal coding standards. “Never use guard clauses.”

Project Rules

Project-specific gotchas. “expect_worker uses lambdas, not kind_of().”

Project Memory

Patterns and observations. “Tests with Time.zone.now fail near midnight.”

Private vs Committed

Global CLAUDE.md, global rules, and project memory are user-private (direct edits). Project CLAUDE.md and project rules are committed to the feature branch via PR.

Two axes per skill — Commit (diff approval) and Continue (phase advancement). Four preset levels from fully manual to fully autonomous. Override any skill at runtime with --auto or --manual. Start cautious, dial up as trust grows.

Part of the Ecosystem

FLOW is part of a growing community of disciplined Claude Code plugins. Two projects worth knowing that inspired and motivated me:

metaswarm by Dave Sifry. A multi-agent orchestration framework with 18 specialized agents, cross-model adversarial review, and full pipeline orchestration from GitHub issue to merged PR. If FLOW is disciplined feature development, metaswarm is disciplined development at scale.

Superpowers by Jesse Vincent. Foundational agentic skills for Claude Code including TDD, systematic debugging, and plan writing. Proved that disciplined agent workflows are not overhead — they’re what make autonomous development reliable.
