GitHub Changelog

FLOW

An opinionated 9-phase software development lifecycle for Claude Code — built for Ruby on Rails. Every feature follows the same phases in the same order. No shortcuts.

Why

Claude Code is powerful, but undisciplined by default. Left to itself, it writes code, skips tests, misses edge cases, and moves on. FLOW imposes structure. Not bureaucracy — discipline.

But there’s a second problem. Engineers, managers, and tech leads are watching AI transform software development and trying to figure out where they fit. One-shot “vibe coding” — describe the feature, let Claude do everything, merge what comes out — feels like giving up control. But manually reviewing every line of AI-generated code doesn’t scale either.

FLOW is for the engineer who wants to stay in the driver’s seat: who decides what gets built, approves the design, reviews the plan, and merges the result — without writing the code themselves. Every phase requires your approval before the next one starts. You remain the engineer. Claude does the work.

How

Every feature goes through a 9-phase orchestration. Each phase has mandatory entry guards, structured outputs written to a persistent state file, and back-navigation when something was missed. No phase can be skipped.

1
Start Worktree · push branch · PR · bin/ci baseline · bundle update · bin/ci green sub-agent
2
Research Full class hierarchy · callbacks · test/support/ · risks documented sub-agent
3
Design 2–3 alternatives validated · user picks one · approved before any code sub-agent
4
Plan Task feasibility verified · TDD ordering · section-by-section approval sub-agent
5
Code Test-first per task · diff review · bin/ci gate · commit per task
6
Review Design alignment · research risk coverage · Rails anti-patterns sub-agent
7
Security Vulnerability scan · auth gaps · data exposure · injection risks sub-agent
8
Reflect Learnings routed to 5 destinations · worktree memory rescued · plugin gaps noted
9
Cleanup Worktree removed · state file deleted · feature done

Phases 2, 3, 4, 6, and 7 use mandatory Explore-type sub-agents to read the codebase. Phase 1 uses a general-purpose Sonnet sub-agent to fix CI failures automatically — whether from a dirty main branch, gem upgrade breakage, or flaky tests. The main conversation stays focused on decisions while sub-agents handle the heavy lifting.

Architecture

Sub-Agent Architecture

Research, Design, Plan, Review, and Security launch Explore sub-agents. Start launches a Sonnet sub-agent to fix CI failures. By the time Code begins, every callback has been found, every risk documented. Code trusts the state file.

Smart Model Selection

Each phase automatically selects the right model via skill frontmatter. Opus for Design, Code, and Security where reasoning matters most. Sonnet for structured phases. Haiku for mechanical steps. CI failures in Start are delegated to a Sonnet sub-agent.

State File Persistence

Every feature has a .flow-states/<branch>.json file tracking research findings, design decisions, plan tasks, captured notes, and per-phase timing. State survives session breaks and compaction.

Session Hook — Auto-Continue

Every session start — new terminal, /clear, /compact — triggers a hook that scans for in-progress features. Claude’s first action is /flow:continue. Close your laptop, open Claude Code the next morning, pick up exactly where you left off.

The Learning System

User corrects Claude → /flow:note captures it → Claude writes observations to worktree memory → Reflect reads all four sources → each learning routed to the right destination (global CLAUDE.md, project CLAUDE.md, global rules, project rules, or project memory). Nothing is lost when the worktree is destroyed.

Phase Back-Navigation

Not a linear checklist. If Design reveals a gap in Research, go back. If Code reveals a flaw in the Plan, go back. Prior findings are preserved and extended — never discarded. Later phases are invalidated and re-approved.

100% Coverage Enforced

Code cannot transition to Review until coverage/uncovered.txt is empty. bin/ci must be green before every commit and every phase transition. Technical debt is blocked at the source.

Minimal Project Footprint

FLOW configures workspace permissions in .claude/settings.json (via /flow:init, committed once). During active development, a single gitignored .flow-states/<branch>.json per feature exists. Cleanup deletes even that. No scripts, no build tools, no runtime dependencies.

Instructions Are Advisory. Gates Aren’t.

Most agent workflows put enforcement in instructions: “always run bin/ci”, “never skip Research”. Instructions work until they don’t. FLOW’s phase enforcement is layered and deterministic. There is no instruction path from an incomplete phase to the next one running.

Inline Phase Guard

Every phase skill opens with a Python gate that reads the state file and exits immediately with BLOCKED if the previous phase isn’t complete. The skill doesn’t run — there’s nothing for Claude to interpret or override.

check-phase.py

A standalone verification script callable from anywhere in the workflow. One source of truth for phase state, shared by skills, hooks, and utility commands alike. No duplication, no drift.

SessionStart Hook

Fires on every session start — new terminal, /clear, /compact. Reads the state file and injects the current phase directly into Claude’s context. After a week away, Claude opens knowing exactly where it is and cannot proceed as if it doesn’t.

What It Enforces

  • Worktree isolation — main is never touched directly; multiple features run in parallel
  • Research before design — full class hierarchy read, callbacks found, risks documented first
  • Design alternatives required — 2–3 distinct approaches validated before user picks one
  • TDD always — test must fail before implementation is written; test must pass before commit
  • bin/ci gate — must be green before every commit and every phase transition
  • 100% test coverage — Code cannot transition to Review until coverage is complete
  • No disabling RuboCop — fix the code, not the cop; no # rubocop:disable comments
  • Commit discipline — imperative verb + tl;dr + per-file breakdown, every commit

Utility Commands

Available at any point in the workflow, regardless of phase.

The Learning System

Most AI coding tools forget everything when the session ends. FLOW captures corrections as they happen, rescues session memory before worktrees are destroyed, and routes each learning to the right permanent home — where it compounds across every future feature.

1
Capture User corrects Claude → /flow:note captures it automatically
2
Observe Claude writes patterns and observations to worktree memory during feature work
3
Synthesize Reflect reads all four sources: state file · notes · conversation · worktree memory
4
Route Each learning is routed to its correct permanent destination
5
Survive Worktree is destroyed — nothing is lost

5 Destinations

Global CLAUDE.md

Process rules that apply to all projects. “Always run bin/ci before committing.”

Project CLAUDE.md

Project-specific architecture. “Callbacks set fields from Current — don’t pass explicitly.”

Global Rules

Universal coding standards. “Never use guard clauses.”

Project Rules

Project-specific gotchas. “expect_worker uses lambdas, not kind_of().”

Project Memory

Patterns and observations. “Tests with Time.zone.now fail near midnight.”

Private vs Committed

Global CLAUDE.md, global rules, and project memory are user-private (direct edits). Project CLAUDE.md and project rules are committed to the feature branch via PR.

Part of the Ecosystem

FLOW is part of a growing community of disciplined Claude Code plugins. Two projects worth knowing that inspired and motivated me:

metaswarm by Dave Sifry. A multi-agent orchestration framework with 18 specialized agents, cross-model adversarial review, and full pipeline orchestration from GitHub issue to merged PR. If FLOW is disciplined Rails development, metaswarm is disciplined development at scale.

Superpowers by Jesse Vincent. Foundational agentic skills for Claude Code including TDD, systematic debugging, and plan writing. Proved that disciplined agent workflows are not overhead — they’re what make autonomous development reliable.

Get Started

All 9 phase skills, state file infrastructure, session hooks, and reference documentation in one plugin.

Step 1: Install in Claude Code

/plugin marketplace add benkruger/flow
/plugin install flow@flow-marketplace

Step 2: Initialize in your Rails project (once per project)

/flow:init

Step 3: Start a feature

/flow:start invoice pdf export

FLOW creates a git worktree, opens a GitHub PR, runs bin/ci to establish a baseline, upgrades gems, runs bin/ci again to confirm green, and lands you in Phase 2: Research.

The plugin installs into Claude Code’s managed plugin directory. /flow:init commits workspace permissions in .claude/settings.json and a version marker in .flow.json (once per project). During active development, a single gitignored .flow-states/<branch>.json per feature tracks state. Cleanup deletes even that. Three commands to set up. One file while you work. Zero when you’re done.
